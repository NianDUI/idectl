# 01 · 需求与范围

> IdeaBridge：一个 IntelliJ IDEA 插件，内嵌 MCP（Model Context Protocol）服务器，把 IDE 的**构建、运行/调试、控制台日志、Maven** 能力开放给外部 AI Agent（Claude Code、Codex 等），并支持**多用户（多 Agent 客户端）多角色**接入。
>
> 本文档定义"做什么、为谁做、做到什么程度、明确不做什么"。架构见 [02-总体架构](02-总体架构.md)，工具 API 见 [03-MCP工具API规范](03-MCP工具API规范.md)。
>
> **核心优先级（2026-07 定稿）**：① 重新运行正在运行的程序，且支持 run/debug 模式切换（FR-RUN-4/6 + `restart_session(mode)`）；② 检索正在运行程序的控制台日志（FR-LOG-1~5 + `console_search`）；③ 热加载：debug 运行中把"简单修改"（仅方法体内改动，不涉及签名/增删方法/增删类）重新构建后**免重启**换进目标 JVM（FR-HOT-1~3 + `reload_classes`）。①②随 M1 首发、③随 M2 首发（见 [09-实现路线图](09-实现路线图与里程碑.md)），其余能力按需后置。

## 1. 背景与动机

AI 编码 Agent（如 Claude Code）目前只能通过 shell 驱动构建与运行（`mvn compile`、`java -jar` 等），与开发者实际使用的 IDEA 之间存在割裂：

- **构建不一致**：IDE 的增量编译（JPS）、注解处理、模块 classpath 与命令行 Maven 不完全一致；Agent 用 CLI 编译"通过"不代表 IDE 里没有红字，反之亦然。
- **运行配置不可达**：开发者维护在 IDE 里的 Run/Debug Configuration（Spring Boot 的 VM 参数、active profiles、JDK 选择、"缩短命令行"策略、Remote JVM Debug 等，见截图 2）对 Agent 完全不可见，Agent 只能自己拼命令行，经常拼错。
- **日志不可读**：Agent 启动的服务日志在 IDE 控制台里，Agent 看不到；Agent 自己启动的进程 IDE 又管不到。
- **调试全靠猜**：Agent 无法打断点、看变量，只能靠加 `System.out` 反复重跑。

IdeaBridge 让 Agent 用与开发者**完全相同**的构建/运行/调试通道工作，同时把控制台变成可增量读取、可 grep 的数据源。

### 1.1 与既有原型（v0）的关系

仓库中已存在一个 v0 原型（见根目录 README：HTTP + MCP，8 个工具）。本套文档是**推倒重来的完整设计**，不受 v0 代码约束；v0 只作为可行性证据（内嵌 HTTP 服务器、ProjectTaskManager、ExecutionListener 均已跑通过）。

## 2. 目标用户与角色

### 2.1 目标用户

| 用户 | 使用方式 | 典型诉求 |
| --- | --- | --- |
| AI Agent（Claude Code / Codex / 自研自动化） | MCP 客户端，通过工具调用 | 编译并拿到结构化错误；启动/停止服务；读日志定位问题；打断点看变量 |
| 人类开发者（插件安装者） | 插件设置页、状态通知、审计日志 | 一次配置、长期无感；能控制 Agent 权限；能追溯 Agent 干了什么 |
| CI/脚本（次要） | 直连 HTTP/MCP 接口 | 复用 IDE 构建做快速本地验证 |

### 2.2 多用户多角色（核心需求）

"多用户"指**多个 Agent 客户端并发连接同一个 IDE**（例如：一个 Claude Code 会话在改代码，另一个在跑回归测试；或人类开发者手动操作 IDE 的同时 Agent 在读日志）。要求：

- 每个客户端是一个**会话（session）**，有独立身份（token）与生命周期；
- 会话可绑定到**指定项目**（同一 IDE 窗口可打开多个项目，见截图 2/4 的多项目多配置场景），未授权项目不可见；
- 按**角色**分级授权（详见 [04-多用户多角色与安全设计](04-多用户多角色与安全设计.md)），基线三级：
  - `viewer`（只读）：列配置、读日志、查构建状态，不能改变任何 IDE 状态；
  - `operator`（操作者）：+ 构建、启动/停止**自己启动的**会话、Maven 同步、断点与单步；
  - `admin`（管理员）：+ 停止任意会话（含人类手动启动的）、表达式求值（≈任意代码执行）、任意 Maven goal、跨项目操作；
- 所有改变状态的调用写入**审计日志**（谁、何时、哪个工具、什么参数、结果）。

编号规则：`FR-<域>-<序号>`。优先级：P0 必须有 / P1 应该有 / P2 可以有。

### 3.2 运行与调试（FR-RUN / FR-DEBUG）— 对应截图 2、3

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| FR-RUN-1 | 列出项目全部 Run/Debug Configuration：名称、类型（Spring Boot/JUnit/Maven/Tomcat/Remote JVM Debug…）、所属文件夹、关键参数摘要（主类、模块、JVM 参数、active profiles） | P0 |
| FR-RUN-2 | 按名称以 **run** 或 **debug** 模式启动配置；返回执行会话 ID | P0 |
| FR-RUN-3 | 启动语义可选：立即返回（长驻服务）或等待进程退出并附退出码与输出尾部（跑测试/一次性任务） | P0 |
| FR-RUN-4 | 停止、重启指定执行会话 | P0 |
| FR-RUN-5 | 列出所有执行会话（运行中 + 最近已结束），含启动方（哪个 Agent 会话 / 人类手动）、执行器（run/debug）、存活状态、退出码 | P0 |
| FR-RUN-6 | **人类手动点绿色按钮启动的进程也被跟踪**（日志可读、状态可查；停止它需要 admin） | P0 |
| FR-RUN-7 | 同名配置多实例（允许并行运行时）能区分 | P1 |
| FR-RUN-8 | 程序化创建临时运行配置（任意 main class / JUnit 类或方法） | P2 |
| FR-DEBUG-1 | 断点：增/删/列（文件+行，支持条件断点） | P0 |
| FR-DEBUG-2 | 调试会话控制：pause / resume / stepOver / stepInto / stepOut | P0 |
| FR-DEBUG-3 | 暂停时读取：线程列表、调用栈帧、指定帧的变量（名称/类型/值摘要，支持按路径展开子字段） | P0 |
| FR-DEBUG-4 | 在指定帧上对表达式求值（**admin 专属**，≈任意代码执行） | P1 |
| FR-DEBUG-5 | 命中断点时的事件可查询（"当前是否暂停、停在哪个 file:line"） | P0 |
| FR-DEBUG-6 | JUnit 会话返回结构化测试结果（用例树：通过/失败/耗时/失败堆栈） | P1 |
| FR-HOT-1 | 热加载（HotSwap）：debug 会话中重新编译修改的类并热替换进目标 JVM，**无需重启**（= 用户习惯：在修改的类上"重新构建"后 IDE 自动重载） | P0 |
| FR-HOT-2 | 能力边界诚实：仅支持方法体内改动（JDWP redefineClasses 标准限制）；签名变更/增删方法/增删类 → 结构化失败并建议 `restart_session(mode=debug)` | P0 |
| FR-HOT-3 | 热加载结果结构化：reloaded / nothing_to_reload / failed（附编译诊断）/ cancelled | P0 |

## 4. 非功能需求（NFR）

| 编号 | 需求 |
| --- | --- |
| NFR-1 **安全** | 仅绑定 `127.0.0.1`；默认要求 Bearer token；求值/任意 goal 等危险能力默认关闭需显式开启；不在日志中回显 token |
| NFR-2 **线程正确性** | 任何 MCP 请求处理不得阻塞 EDT；对 IntelliJ 模型的读写遵循 ReadAction/WriteAction 规则；长操作可取消 |
| NFR-3 **兼容性** | sinceBuild=233（2023.3）起，无 untilBuild 上限；Maven 能力为可选依赖，缺失不影响其余功能 |
| NFR-4 **性能** | 日志增量读取 P50 < 50ms；grep 在 10 万行缓冲内 < 500ms；缓冲默认每会话 ≤ 2 万行 / 8 MB，全局 ≤ 128 MB；已结束会话默认保留最近 20 个 |
| NFR-5 **稳定性** | 单个 Agent 会话异常不影响服务器与其他会话；插件卸载/IDE 退出时干净释放端口与进程监听 |
| NFR-6 **可观测性** | 插件自身日志走 IDE log；审计日志独立落盘；健康检查可探活 |
| NFR-7 **Token 经济性** | 所有工具返回有大小上限与截断标记；列表类支持分页；错误信息可直接行动（含建议） |

## 5. 明确不做（Non-Goals）

1. **不做文件读写/代码编辑工具**——Claude Code 自身与 JetBrains 内建 MCP 已覆盖，避免重复与越权面扩大；
2. **不做 Git 操作**；
3. **不做跨机器远程访问**（仅 localhost；远程需求由用户自行做 SSH 隧道）；
4. **不绕过 IDE 直接调 mvn/gradle CLI**——那是 Agent 用 shell 就能做的，本插件的价值恰在"走 IDE 通道"；
5. **不做图形界面自动化**（不模拟点击 UI）；
6. Spring Boot 运行面板的 Bean/映射/运行状况等专有 tab（截图 3）**首期不做**，列为远期观察项；
7. 首期不做 Gradle 深度集成（构建委托场景按 FR-BUILD-9 尽力兼容，Gradle 工具窗口级功能不做）。

## 6. 关键场景（验收视角）

1. **修代码→编译→看错误闭环**：Agent 改完代码调 `build`（增量），2 秒内拿到 `errors: [{file, line, message}]`，据此继续修改——全程不碰 shell。
2. **起服务→等就绪→读日志**：Agent 以 debug 模式启动 `Application`（截图 2 的 Spring Boot 配置），用 `wait_for_pattern("Started Application")` 等就绪，再 grep `ERROR|Exception` 检查启动质量。
3. **断点调试**：Agent 在 `OlympiaDomain.java:120` 打条件断点，重启会话复现请求，命中后读第 0 帧变量确认字段值，删断点、resume。
4. **Maven 依赖变更**：Agent 改了 `pom.xml`，调 `maven_sync` 等待同步完成，同步失败时直接拿到失败原因文本。
5. **双 Agent 并行**：Agent A（operator，绑定项目 hotel-facai）跑测试；Agent B（viewer）同时 tail 生产联调服务日志；B 尝试停止 A 的进程被拒（403 + 说明）；人类在 IDE 手动启动的进程 B 可读日志。
6. **退出码 137 排查**（截图 3 场景）：进程被 SIGKILL，Agent 通过 `list_executions` 看到 `exitCode=137`，grep 内存相关日志，调大 `-Xmx` 建议交给用户。
7. **热加载循环**（核心③）：`Application` 正以 debug 跑着，Agent 改了某方法的内部逻辑 → `reload_classes(files=["OlympiaDomain.java"])` → 秒级生效、进程不重启（pid 不变）；若 Agent 新增了方法 → 返回 `failed` +"结构性变更需重启"→ Agent 转 `restart_session(mode=debug)`。

## 7. 术语表

| 术语 | 含义 |
| --- | --- |
| **Agent 会话（client session）** | 一个 MCP 客户端连接（含身份/角色/项目范围），区别于"执行会话" |
| **执行会话（execution）** | 一次运行/调试进程实例（对应 IDE Run/Debug 工具窗口的一个 tab），有唯一 `executionId` |
| **RunConfiguration** | IDE 的运行/调试配置（截图 2 左侧树的一项） |
| **增量构建 / 重新构建** | 对应 IDE 的 Build（增量）与 Rebuild（clean 全量） |
| **同步（Maven sync/reimport）** | 让 IDE 项目模型与 pom.xml 重新对齐（截图 4 的"全部刷新"） |
| **JPS** | JetBrains 内置构建系统（非委托构建时实际执行编译者） |
| **热加载（HotSwap）** | 经 JDWP `redefineClasses` 把重新编译的字节码换进运行中的 JVM，仅限方法体内改动，debug 会话专属 |
