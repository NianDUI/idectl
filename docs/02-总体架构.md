# 02 · 总体架构

> 本文是 IdeaBridge 的架构定稿。由三份独立架构提案（平台稳健性 / 多智能体安全治理 / Agent 工效学三视角）综合而成，所有平台 API 结论以 `docs/research/01~06` 的一手调研为准。
>
> **核心优先级**（2026-07 定稿）：① 重新运行正在运行的程序（run/debug 可切换）② 检索正在运行程序的控制台日志。架构必须首先为这两条服务，其余能力在同一骨架上渐进叠加。

## 1. 一张图


## 2. 关键决策记录（ADR）

每条决策注明依据（R01~R06 = `docs/research/` 六份报告）。

| # | 决策点 | 定稿 | 核心理由 | 主要备选（否决原因） |
| --- | --- | --- | --- | --- |
| D1 | 内嵌服务器与 SDK | 插件自带 **Ktor CIO** embeddedServer + 官方 **MCP Kotlin SDK**（锁版本，全部 SDK 类型封在 `McpTransportAdapter` 后不外泄） | 与 JetBrains 2025.2+ 内建 MCP 同构（官方已蹚坑）；协程模型与平台一致（R05） | Java SDK（引入 Reactor/Jackson，与平台协程割裂）；IDE 内建 RestService 63342（FullHttpRequest 缓冲模型不宜 SSE 长流、有信任弹窗，仅留作发现端点）；JDK HttpServer（作降级预案） |
| D2 | 传输协议 | 仅 **Streamable HTTP**（单端点 `/mcp`），协议基线 2025-11-25，向下协商 2025-03-26/06-18；**不做 legacy SSE** | Codex 不支持 SSE、Claude Code SSE 已弃用（R05） | SSE 双栈（纯维护成本） |
| D3 | 端口与多实例 | 端口 **48620** 起 `findFirstFreePort` 递增（避开内建 MCP 的 64342±240 段）；每实例写 `~/.idea-bridge/instances.json`（原子写 + PID/端口双探活清僵尸），条目含 `{port, pid, ide, projects[], tokenHint}` | 规避 JetBrains 全局设置端口覆写缺陷（IJPL-207839）；离线可读、零 mDNS 依赖（R05/R06） | 端口扫描（多实例连错）；mDNS（客户端生态零支持） |
| D4 | 项目路由 | **混合**：MCP 会话默认绑定项目（单项目 IDE 自动绑定；`bind_project` 显式绑定；stdio 桥经 env 绑定），所有工具的 `project` 参数为**可选覆盖**；无论哪条路，最终生效项目必须 ∈ token.allowedProjects，解析集中在 ToolGate 单点 | 兼顾"不同项目路由给不同 Agent"（绑定即隔离）与单连接跨项目灵活性；校验单点化避免安全方案担心的"分散校验被遗漏" | 纯每工具必填 project（token 浪费、LLM 常填错路径——JetBrains 内建即反面教材）；纯绑定（多项目 Agent 被迫多连接） |
| D5 | 长操作语义 | **同步优先、句柄兜底**：build/maven_sync/等待退出默认 `wait=true` + timeout；超时**不算失败**，返回 `operationId` 转 `get_operation(wait_sec)` 长轮询；断线≠取消，显式 cancel 才取消（best-effort） | Agent 主流交互一问一答；超时返回句柄避免 LLM 把慢构建误判为失败而重试（三案一致） | 纯异步轮询（往返翻倍）；MCP tasks 扩展（仍实验性，接口形状已对齐便于将来迁移）；无限阻塞（饿死协程池） |
| D6 | 控制台数据源 | **自建环形缓冲**，在 `ExecutionListener.processStarting(env, handler)` 挂 `ProcessListener`（此回调发生在 `startNotify()` 之前，零丢失，R03）；**不读** ConsoleViewImpl（impl 包不稳定、200ms 延迟 flush、1MB cycle buffer 静默截断、需 ReadAction） | 读路径完全脱离平台锁——稳健性最大红利；晚挂会话用 Document 一次性抢救导入并标记 `attachedLate` | 读 ConsoleView Document 为主（全面劣势）；落盘临时文件（无 offset 协议与清理契约） |
| D7 | 缓冲参数与溢出 | 每会话 **10000 行且 8MB** 双上限（先到为准）、单行 16KB 截断；全局 **128MB** LRU（优先淘汰终止最久会话）、终止会话缓冲 TTL 30 分钟；offset 全局单调**不回退**，越界读取不报错而返回 `gap=true + firstAvailableOffset`（Kafka 语义） | Agent 永远拿到确定性答案而非异常；量级对齐 IDEA 1MB cycle buffer / GHA ~4MB 并留裕量（三案一致） | 无上限（内存失控）；报错（Agent 无法自恢复） |
| D8 | 权限模型 | 预共享 **Bearer token → Principal{subject, role, allowedProjects}**；三级角色 `viewer / operator / admin` + 资源属主动态检查 + 策略开关（求值三档、Maven goal 白名单）；`tools/list` 按角色过滤（低角色**看不见**危险工具）；默认必须有 token、危险能力默认关 | 三档覆盖"只读观察者/干活 Agent/运维 Agent"真实分工，配置 5 秒能看懂；比内建 MCP 的 brave-mode 单开关细一个量级（R06） | OAuth 2.1 全套（localhost 场景过重，规范中本就 OPTIONAL，接口形状已预留）；逐调用人工确认（无人值守不可行，仅留作可选 elicitation 开关） |
| D9 | 工具面聚合 | **一个意图一个工具，动作用 enum 收敛**：四种构建合一 `build(scope, mode)`；步进类合一 `debug_control(action=…)`；断点增删列合一 `breakpoints`；`console_read` 内置 wait_for_pattern/exit；`run_configuration` 内置 wait+ready_pattern+tail。总计 ~27 个 | LLM 工具选择准确率随工具数下降；"启动 Spring Boot 并确认就绪"从 4 次往返压到 1 次 | 一动作一工具（40+，膨胀近一倍）；单一万能 execute（schema 失去约束力） |
| D10 | 构建诊断收集 | **双通道**：`BuildViewManager.addListener` 收 `FileMessageEvent`（跨构建系统主通道，FilePosition 0 基→对外统一 1 基）＋ `CompilerTopics.COMPILATION_STATUS` 收 `CompileContext.getMessages`（JPS 交叉校验）；成败以 `Promise<Result>` 判定，绝不嗅探文本 | `Result` 无错误明细；委托 Gradle/Maven 构建时 JPS 监听全部静默，单通道必漏（R01） | 仅 CompilerTopics（委托构建全盲）；解析 Build 窗口文本（不可靠） |
| D11 | 构建并发 | ProjectTaskManager 本身不互斥（R01），自建**每项目构建 FIFO 队列 + Mutex**（深度 4，满则 BUSY）；run 的 before-launch 构建并入同一把锁；取消 best-effort 且仅发起者/admin | 隐式 JPS 信号量只会无反馈阻塞；队列位置可观测（`queued + queuePosition`） | 直接放行并发（行为不可预期）；拒绝第二个请求（Agent 协作体验差） |
| D12 | 与 JetBrains 内建 MCP 关系 | **独立共存 + 差异化**：不依赖 `com.intellij.mcpserver.*`（2025.2 才存在，我们 sinceBuild=233）；端口错开；不做官方已做好的文件/搜索/重构工具；主打其实证空白——会话生命周期、控制台增量+grep、手动会话跟踪、构建粒度与诊断、Maven 全套、多客户端角色治理（R06 差距表：20 项核心能力 12 项全生态空白） | 233~2025.1 是独占区间；官方通道无认证无会话概念，治理只能在自有通道实现 | 寄生官方 mcpToolset EP（放弃独占区间与治理；2025.2+ 可选注册只读子集作引流，M6 再评估） |
| D13 | 依赖与类加载 | 自带 Ktor/kotlin-sdk/serialization，但**构建期断言产物 zip 无任何 kotlinx-coroutines jar**（用平台捆绑补丁版）；Maven 集成全部类型隔离在 optional descriptor（`depends optional=true` + config-file），核心 classloader 零 `org.jetbrains.idea.maven` 引用 | 社区版 coroutines 混入会产生诡异挂起 bug；Maven 禁用时避免 NoClassDefFoundError（R04/R05） | Shadow relocate（ServiceLoader/反射风险，留作冲突后手） |
| D14 | 行列号约定 | 对外**一律 1 基**（文件行、断点行、诊断行列）；平台 0 基值（FilePosition、XLineBreakpoint.line）在适配层边界统一转换，且写入每个工具 description | LLM 与编辑器习惯 1 基；不统一必出 off-by-one | 透传平台值（Agent 侧混乱） |
| D15 | 重启语义（核心①） | `restart_session(session_id, mode: same\|run\|debug)`：`same` 走 `ExecutionUtil.restart(descriptor)`；**切换模式**走 stop（soft kill→wait→force 兜底）→ 取原会话的 `RunnerAndConfigurationSettings` → `ExecutionEnvironmentBuilder.create(目标 executor, settings).buildAndExecute()`；新旧会话经 `restartOf` 字段关联 | 用户核心需求："把正在 run 的进程直接以 debug 重启"是高频调试入口，竞品全部缺失 | 仅 same-executor restart（不满足需求） |
| D16 | 热加载通道（核心③） | 复用 Java 调试器官方 HotSwap 设施：`com.intellij.debugger.ui.HotSwapUI.reloadChangedClasses(session, compileBeforeHotswap, HotSwapStatusListener)`；指定文件时组合"`ProjectTaskManager.compile(files)` → reload"（compileAndReload 无 listener 重载，组合才能同时拿到编译诊断与热替换结果）；能力边界=标准 JDWP redefineClasses（仅方法体内改动），结构性变更诚实失败并建议 `restart_session(mode=debug)` | 与 IDE"修改类上重新构建→自动重载"行为完全一致（用户既有心智）；带回调重载可编程免弹窗；不造轮子 | 裸 JDWP 自实现 redefineClasses（重复造轮子且绕开 IDE 编译产物定位）；Spring devtools（本质是重启 context，不满足"不重启"）；DCEVM/JBR 增强热替换（依赖特定 JVM，列观察项） |

## 3. 模块结构（单插件工程 + 两个附属产物）

```
idea-bridge/
├─ src/main/kotlin/com/ideabridge/
│  ├─ transport/        # Ktor server、McpTransportAdapter、AuthInterceptor（D1/D2）
│  ├─ session/          # SessionManager、Principal、TokenStore（PasswordSafe 存密文）
│  ├─ gate/             # ToolGate：角色/项目路由/属主/配额单点校验 + 审计埋点（D4/D8）
│  ├─ tools/            # ToolRegistry + 各域工具定义（inputSchema/outputSchema/annotations）
│  ├─ core/
│  │  ├─ exec/          # ExecutionRegistry、会话属主、restart 编排（D15）
│  │  ├─ console/       # ConsoleRingBuffer、SearchEngine（deadline-grep）、AnsiEscapeDecoder 接线
│  │  ├─ build/         # BuildService：队列/互斥/双通道诊断聚合（D10/D11）
│  │  ├─ debug/         # DebugService：XDebugger 桥接（suspendCancellableCoroutine+withTimeout）
│  │  ├─ test/          # TestResultCollector（SMTRunnerEventsListener）
│  │  ├─ op/            # OperationRegistry：长操作句柄（D5）
│  │  └─ audit/         # AuditService（JSONL 按日滚动）、QuotaService
│  ├─ maven/api/        # MavenFacade 接口（核心 classloader 只见接口，D13）
│  ├─ settings/         # 设置页：端口/token 签发吊销/角色/白名单/缓冲参数
│  └─ discovery/        # instances.json 注册表（D3）
├─ src/main/resources/META-INF/
│  ├─ plugin.xml               # sinceBuild=233；depends com.intellij.java；projectListeners 声明式注册
│  └─ ideabridge-maven.xml     # optional descriptor：MavenFacade 实现 + Maven 工具注册（D13）
├─ maven-impl 源集             # MavenFacadeImpl（import org.jetbrains.idea.maven.* 仅限此处）
└─ stdio-bridge/               # 独立 thin JAR：stdio←→HTTP 转发，读注册表按项目路径最长前缀选实例
```

依赖方向自上而下单向；`core/*` 不 import Ktor 与 SDK 类型；平台 API 调用集中在各服务内部的薄适配函数，便于版本破裂时只改一处。

## 4. 线程与协程模型（NFR-2 的落实）

| 场景 | 规则 |
| --- | --- |
| tool handler | `suspend fun`，跑 `Dispatchers.Default`；**禁止**直接触碰 PSI/VFS/EDT |
| 读 IDE 模型（配置列表、模块树） | `readAction { … }` |
| 启动/停止/重启运行配置、断点增删 | `withContext(Dispatchers.EDT)`（断点写操作再包 `WriteAction`，兼容 233~最新，R02） |
| `Promise`（ProjectTaskManager 等） | 一律 `.await()`；**全库禁用 `blockingGet`**（EDT 上必死锁——Promise 在 EDT resolve，R01），加 lint 检查 |
| 回调式异步 API（XDebugger 帧/变量/求值、SM 测试） | 统一模板：`suspendCancellableCoroutine + withTimeout`；"回调可能永不到达"按超时返回结构化错误 |
| 平台监听器回调（EXECUTION_TOPIC / COMPILATION_STATUS / BuildProgressListener） | 全部 try/catch（`syncPublisher` 异常会传播回构建流程）；只做 O(1) 登记后投递 Channel，重活在协程里干 |
| 控制台读/检索 | 只碰自建缓冲的锁，**零平台锁** |
| 长操作 | `withBackgroundProgress` + 进度转 MCP `notifications/progress`；MCP cancel → 协程 cancel → ProgressIndicator |
| 生命周期 | app 级 `@Service` 构造注入平台 CoroutineScope；每 MCP 会话独立 `SupervisorJob` 子作用域（故障隔离）；`dispose()`+`beforePluginUnload` 内 `NonCancellable + withTimeout(2s)` 同步停服务器、注销注册表（动态卸载无 classloader 泄漏） |

## 5. 核心场景数据流

### 5.1 核心②：检索正在运行程序的控制台日志

```
IDE 启动（projectListeners 声明式注册，不依赖插件代码先跑）
  → 用户手点绿色按钮 / Agent 调 run_configuration
  → EXECUTION_TOPIC.processStarting(executorId, env, handler)   ← startNotify 之前，零丢失
      ExecutionRegistry 登记 {sessionId(UUID), configName, typeId, executor, owner, startedAt}
      handler.addProcessListener → onTextAvailable(text, outputType)
        → AnsiEscapeDecoder(每会话实例) 剥离 ANSI → ConsoleRingBuffer.append(line)  [O(1)]
  → Agent: console_search(session_id, pattern="ERROR|Exception", after=3)
      ToolGate(viewer 即可) → SearchEngine：ERE 正则 + deadline-CharSequence(500ms 熔断)
      → {hits[{line, before[], after[], matchRanges}], totalScanned, gap}   ← 不碰任何平台锁
```

插件晚装/晚启用的兜底：`ExecutionManager.getRunningProcesses` 补挂监听 + ConsoleView Document 一次性抢救导入，会话标记 `attachedLate=true`、`firstAvailableOffset>0`——对 Agent 诚实说明"更早的日志不可考"。

### 5.2 核心①：重新运行（可切换 run/debug）

```
Agent: restart_session(session_id=…, mode=debug, wait=ready, ready_pattern="Started .* in")
  → ToolGate：属主检查（自己的会话=operator；他人/human 的=admin）→ 审计
  → ExecutionRegistry.find(sessionId) → 取其 ExecutionEnvironment / RunnerAndConfigurationSettings
  → mode==same → withContext(EDT) { ExecutionUtil.restart(descriptor) }
    mode!=same → stop 编排：destroyProcess(soft) → waitFor(timeout) → force 兜底
               → withContext(EDT) { ExecutionEnvironmentBuilder
                     .create(project, DefaultDebugExecutor, settings).buildAndExecute() }
  → processStarting 自动进注册表（restartOf=旧 id）→ 按 wait 语义等就绪/退出
  → 返回 {sessionId(新), restartOf, state, tail[…], nextOffset, debugSessionId?}
```

单实例配置（`allowRunningInParallel=false`）冲突：启动前预检 running descriptor，返回结构化 `CONFLICT`（附现存 sessionId 与建议动作），**绝不触发平台模态弹窗**卡死请求。

### 5.3 构建（M2 起）

```
build(scope=module, modules=["hotel-facai-core-domain"], wait=true)
  → BuildService 队列（每项目 Mutex，深度 4）→ 订阅两通道诊断
  → ProjectTaskManager.build(modules).await()
  → {status, errors[{file, line(1基), column, message, module}], durationMs}
  超时 → {status=timeout, operationId} → get_operation(operation_id, wait_sec=30) 续接
```

## 6. 多 IDE 实例与客户端接入

- 每个 IDE 实例一个服务器实例、一个端口；`instances.json` 提供"端口 ↔ 打开的项目路径"映射。
- HTTP 客户端（Claude Code 等）：`claude mcp add --transport http idea-bridge http://127.0.0.1:<port>/mcp --header "Authorization: Bearer <token>"`。
- 仅支持 stdio 的客户端：走 `stdio-bridge` thin JAR，按 `IJB_PROJECT_PATH` 环境变量对注册表做最长前缀匹配自动选实例。
- `get_ide_state` 返回 `pluginVersion + openProjects`，Agent 可自证连对了实例。
