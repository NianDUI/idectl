# 05 · 构建功能设计

> 详细 API 依据见 [research/01-intellij-build-api.md](research/01-intellij-build-api.md)。工具契约见 [03 规范 §5](03-MCP工具API规范.md)。里程碑 M2。

## 1. 入口：ProjectTaskManager（唯一与 Build 菜单对齐且兼容 delegated build 的 API）

| IDE 菜单（截图 1） | 平台 Action | Idectl 调用 | build 工具参数 |
| --- | --- | --- | --- |
| 构建项目 ⌘F9 | CompileDirtyAction | `buildAllModules()` | `scope=project, mode=incremental` |
| 构建模块 'xxx' | MakeModuleAction | `build(vararg Module)` | `scope=module, modules=[…]` |
| 重新编译 'X.java' ⇧⌘F9 | CompileAction | `compile(vararg VirtualFile)` | `scope=files, files=[…]` |
| 重新构建项目 | CompileProjectAction | `rebuildAllModules()` | `scope=project, mode=rebuild` |
| （模块级 rebuild） | — | `rebuild(vararg Module)` | `scope=module, mode=rebuild` |

全部返回 `Promise<ProjectTaskManager.Result>`；**Result 只有 `isAborted/hasErrors`，无错误明细**——诊断必须旁路收集（§2）。Promise 在 EDT resolve，一律 `.await()`，禁 `blockingGet`（EDT 必死锁，R01）。

## 2. 诊断双通道（D10）

```
BuildService.executeBuild(request):
  1. 生成 buildId；注册两通道收集器（先订阅，后发起——规避事件时序竞态）
  2. 通道 A（跨构建系统主通道）：BuildViewManager.addListener(BuildProgressListener, disposable)
       FileMessageEvent → {kind: ERROR|WARNING|INFO|STATISTIC, FilePosition(file, 0基行列), message}
       OutputBuildEvent → 灌入构建输出缓冲（07 §6）
  3. 通道 B（JPS 交叉校验）：CompilerTopics.COMPILATION_STATUS
       CompilationStatusListener.compilationFinished(aborted, errors, warnings, context)
       context.getMessages(CompilerMessageCategory.*) → CompilerMessage
       （接口无行列号，经 Navigatable/OpenFileDescriptor 提取；impl 类 1 基）
  4. ProjectTaskManager.xxx().await() 判定成败
  5. 聚合：按 (file, line, column, message) 去重合并两通道，行列统一 1 基
  6. 结果存 OperationRegistry（断线后 get_operation / get_build_status 可重查）
```

**为什么双通道**：构建委托给 Gradle/Maven 时（`Settings → Build Tools → Build and run using`），JPS 监听（通道 B）**全部静默**，只有通道 A 有事件；JPS 场景通道 B 的 CompilerMessage 语义最精确。两通道并联 + 去重是唯一全场景可靠方案（R01）。

## 3. 并发：每项目构建队列（D11）

- ProjectTaskManager **不互斥**；JPS 由 `CompilerManagerImpl` 单许可信号量隐式串行（无反馈阻塞）。Idectl 自建**每项目 FIFO 队列 + Mutex**：
  - 深度 4，满则 `BUSY`（附当前持有者 subject 与 buildId）；
  - 入队即返回 `status=queued, queuePosition, operationId`；
  - `run_configuration` 的 before-launch 构建**并入同一把锁**——Agent 显式 build 与隐式 build 不打架；
  - 跨项目互不阻塞（注意：JPS BuildManager 是 application 级，极端并发下仍可能共享外部构建进程资源——R01 开放问题，实现期实测）。
- 用户手动菜单构建：经 `ProjectTaskListener.TOPIC` 感知，队列状态上报 `EXTERNAL_BUILD_RUNNING`（不抢锁，只观测）。
- **取消**：无公开取消 API（R01）→ `cancel_operation` 声明 best-effort（JPS 经 ProgressIndicator.cancel 尝试；排队中的直接移除=确定成功）。

## 4. 增量 vs 全量语义

- `incremental`（构建项目/模块）：JPS 增量编译，成本低，Agent 修代码后的默认动作；
- `rebuild`：clean 全量，队列中标注高成本（timeout 默认 600s vs 300s）；
- 自动构建（automake）与手动构建关系：automake 走独立回调（`automakeCompilationFinished`）且不进 Build 窗口，Idectl **不跟踪** automake（边界写入文档），只保证显式构建语义精确。

## 5. 已知坑（实现期核对清单）

1. 行列号基线混乱：`FilePosition` 0 基、`CompilerMessageImpl` 1 基 → 适配层统一 1 基 + 边界测试（R01 开放问题）。
2. `BuildViewManager.addListener` 位于 lang-impl 且 @Experimental；2025.x Remote Dev 拆分出 FrontendBuildViewManager → 收敛进薄适配类，CompilerTopics 兜底，每个 EAP 跑 Plugin Verifier + 诊断回归。
3. 监听注册与 StartBuildEvent 的时序竞态 → 先订阅后发起；对"是否收到本 buildId 的 start 事件"做断言。
4. Kotlin 编译错误在两通道的覆盖度需实测（javac/kotlinc/注解处理器差异，R01 开放问题）。
5. 第二个构建请求在 CompilerManagerImpl 信号量上的实际表现（阻塞线程/Promise 时机）需实测以校准队列参数。

## 6. 与热加载的联动

`reload_classes`（热加载 ★核心③，见 [06 §7](06-运行与调试功能设计.md)，M2）在指定 files 时复用本域 `compile(files)`（⇧⌘F9 语义）与双通道诊断：编译失败直接把结构化错误返回给 Agent、不发起热替换；编译成功才调 `HotSwapUI.reloadChangedClasses`。
