# 07 · 控制台日志读取与检索设计（★核心②：搜索正在运行程序的日志）

> 详细 API 依据见 [research/03-console-capture-and-search.md](research/03-console-capture-and-search.md)。工具契约见 [03 规范 §4](03-MCP工具API规范.md)。

## 0. 为什么不读 IDE 自己的 ConsoleView

`ConsoleViewImpl` 属 impl 包非稳定 API（master 已 Kotlin 化、233→252 签名漂移）、有 200ms 延迟 flush、自带 1MB cycle buffer **静默截断**、读 Document 需 ReadAction（引入平台锁）。自建缓冲让 MCP 读路径**完全脱离平台锁**——这是本设计最大的稳健性红利（R03 全面论证）。ConsoleView Document 仅用于 attachedLate 会话的一次性历史抢救导入。

## 1. 捕获管线

```
processStarting(executorId, env, handler)      ← startNotify() 之前，零丢失（R03 源码验证）
  └─ handler.addProcessListener(object : ProcessAdapter() {
       override fun onTextAvailable(event, outputType) {
         // 快路径必须 O(1)：任何耗时都会拖慢平台输出泵
         val stream = ProcessOutputType 判别         // 用静态方法，ANSI 上色后是"颜色子类型"，禁止引用比较
         ansiDecoder.escapeText(event.text, ...)      // 每会话独立 AnsiEscapeDecoder（跨 chunk 有状态）
           → ringBuffer.append(ts, stream, cleanText) // 加锁 O(1) 入队
           → waiters.signal()                          // 唤醒 wait_for_pattern / wait=ready 的长轮询者
       }
     })
```

- `ColoredProcessHandler` 已内置解码的场景直接取干净文本，避免二次解码。
- `\r` 进度条类输出（Gradle 富控制台）：同 offset 行覆盖策略——保留最后一帧，标记 `progressLine=true`（实现期按 R03 开放问题实测调整）。

## 2. ConsoleRingBuffer 数据结构

```kotlin
class ConsoleRingBuffer(
    private val maxLines: Int = 10_000,       // 与 maxBytes 先到为准
    private val maxBytes: Long = 8 * MB,
    private val maxLineLen: Int = 16 * 1024,  // 单行截断，防单行巨無霸
) {
    // ArrayDeque<Line>；Line = {offset:Long, ts:Long, stream:Byte, text:String, truncated:Boolean}
    // firstOffset = 队头行号；nextOffset = 队尾行号+1 —— 全局单调，淘汰只前移 firstOffset，永不回退
    @Synchronized fun append(...)             // O(1)，溢出 pop 队头
    @Synchronized fun read(from: Long?, maxLines: Int, maxBytes: Int, streams: Set?): ReadResult
    @Synchronized fun snapshotFor(search)     // 检索前抓一致性快照（或持锁分段扫）
}
```

**offset/gap 语义（Kafka 式）**——Agent 永远拿到确定性答案：

| 请求 | 响应 |
| --- | --- |
| `from_offset ∈ [firstOffset, nextOffset)` | 正常增量返回，`gap=false` |
| `from_offset < firstOffset`（已淘汰） | 从 firstOffset 起返回，`gap=true, droppedLines=firstOffset-from_offset` |
| `from_offset ≥ nextOffset`（追平） | 空 lines + 当前 nextOffset（配合 wait 参数变长轮询） |
| 缺省 from_offset | = firstAvailableOffset（读所有还在缓冲里的） |

**全局内存治理**：所有缓冲注册到 App 级账本；总量超 128MB 时 LRU 淘汰"终止最久"会话的整段缓冲；终止会话缓冲保留 30min TTL 后释放（`list_sessions(state=terminated)` 在 TTL 内日志仍可读——FR-LOG-4）。水位暴露在 `get_ide_state`。

## 3. 增量读取（console_read）

- 纯内存操作，持缓冲锁拷贝切片即返回，P50 < 50ms（NFR-4）。
- `max_bytes`（默认 64KB）控制 Agent 的 token 预算，先于 max_lines 触发时置 `truncated`。
- **长轮询合一**：`wait_for_pattern` / `wait_for_exit` + `wait_timeout_sec`：
  - 先查存量（pattern 可能早已出现）；未命中则挂 waiter（Channel），append 时逐行匹配唤醒；
  - 进程退出唤醒所有 waiter（返回 `sessionState=terminated, exitCode`）；
  - 超时返回当前进度（`matched=null`），**不报错**——Agent 拿到 nextOffset 可继续。
  - 典型用法：`run_configuration(wait=ready, ready_pattern="Started .* in .* seconds")` 内部即复用此机制。

## 4. 检索引擎（console_search）★

**语义 = `grep -E -i -A -B -m`**：ERE 正则、大小写开关、before/after 上下文行（≤20）、max_matches（≤1000）、范围过滤（offset 区间 / 时间区间 / stream 过滤）。

```
执行流程：
1. 参数校验：pattern ≤512 字符，编译 Pattern（ignore_case → CASE_INSENSITIVE|UNICODE_CASE）
2. 取快照（持锁抓引用切片，释放锁后扫描——不阻塞高频 append）
3. 逐行 matcher.find()，输入包 DeadlineCharSequence：
     每次 charAt 检查预算，超 500ms 抛 SearchTimeout → 返回已有命中 + truncatedByDeadline=true
     （防 ReDoS：恶意 (a+)+$ 之类的灾难回溯 500ms 内熔断）
4. 命中组装：{line, before[], after[], matchRanges[[start,end]]}；相邻命中的上下文窗口自动合并去重
5. 元数据：totalScanned / truncatedByMaxMatches / gap（缓冲已淘汰部分 → 告知检索不完整）
```

- `matchRanges` 给出行内命中区间，Agent 可高亮/精确摘取。
- 多行匹配（跨行 pattern）**不支持**（v1 明确边界，写入工具 description）；Agent 需要时可 console_read 拉区间自行处理。
- 性能预算：10 万行缓冲 < 500ms（NFR-4）；超时熔断兜底。

## 5. 结构化测试结果（get_test_results，M2）

- 订阅 `SMTRunnerEventsListener.TEST_STATUS`（app 级 Topic），按 ProcessHandler 关联到 ExecutionRecord。
- `SMTestProxy` 树 → `{summary, failures[{name, locationUrl, message, stacktrace(截断)}]}`；默认 `failed_only=true` 省 token。
- 覆盖 JUnit 直跑与 Gradle 委托测试；Maven surefire 直跑 goal 无 SM 事件（R03），此时 Agent 退回 console_search "Tests run:" 文本（工具 description 注明）。

## 6. 构建/同步输出复用（M2/M5）

`BuildViewManager` / `SyncViewManager.addListener(BuildProgressListener, Disposable)` 的 `OutputBuildEvent` 流灌入同一套 RingBuffer 设施（buildId 为句柄）→ `get_operation` 结果附 `outputAvailable=true`，`console_read/search` 以 `build:<buildId>` 形式的 session_id 同样可读可检索（FR-LOG-6）。`OutputBuildEvent.getOutputType()` 在 233 的存在性未证（R03 开放问题）→ 反射探测回退 `isStdOut()`。

## 7. 边界诚实性清单

| 场景 | 返回 |
| --- | --- |
| 插件晚挂（attachedLate） | `attachedLate=true, firstAvailableOffset>0`；抢救导入的历史文本无 stream/ts 标记（stream=system, ts=导入时刻） |
| 缓冲淘汰 | `gap=true + droppedLines` |
| PTY 会话 | `pty=true`（stdout/stderr 混流不可分） |
| 单行超 16KB | `truncated=true` |
| 检索到达 deadline | `truncatedByDeadline=true` + 已得命中 |
| 会话已淘汰（TTL 过） | `NOT_FOUND` + remediation（提示 include_terminated 窗口已过） |
