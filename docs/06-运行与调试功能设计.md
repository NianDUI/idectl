# 06 · 运行与调试功能设计（★核心①：重新运行 run/debug）

> 详细 API 依据见 [research/02-intellij-run-debug-api.md](research/02-intellij-run-debug-api.md)。工具契约见 [03 规范 §3/§6](03-MCP工具API规范.md)。

## 1. 执行会话注册表（ExecutionRegistry）—— 一切的地基

### 1.1 数据模型

```kotlin
class ExecutionRecord(
    val sessionId: String,              // UUID，对外唯一句柄
    val project: Project,
    val settings: RunnerAndConfigurationSettings?,  // 重启的关键；手动会话从 env 取
    val environment: ExecutionEnvironment,          // executionId、executor、runner
    val processHandler: ProcessHandler,
    val descriptor: RunContentDescriptor?,          // ExecutionUtil.restart 需要
    val executor: String,               // "Run" | "Debug"（DefaultRunExecutor/DefaultDebugExecutor.EXECUTOR_ID）
    val owner: String,                  // token.subject 或 "user"
    val startedAt: Long,
    val restartOf: String?,             // 重启链
    val buffer: ConsoleRingBuffer,      // 见 07
    @Volatile var exitCode: Int?,       // processTerminated 回调回填
    @Volatile var state: State,         // RUNNING / TERMINATED
    val attachedLate: Boolean,
)
```

### 1.2 注册来源（双通道，全覆盖）

1. **消息总线（主通道）**：`plugin.xml` 中 `projectListeners` **声明式**订阅 `ExecutionManager.EXECUTION_TOPIC`——不依赖插件代码先跑，IDE 启动即生效，因此**用户手动点绿色按钮的会话同样被捕获**（`owner=user`）。
   - `processStarting(executorId, env, handler)`：唯一正确的挂载点——此回调发生在 `ProcessHandler.startNotify()` 之前（R03 源码验证），在此登记 record + `handler.addProcessListener(...)`，**零丢失首行输出**。
   - `processStarted`：置 state=RUNNING、补 descriptor。
   - `processNotStarted`：must-handle——否则 Agent 的 run 调用悬挂（R02）。
   - `processTerminated(handler, exitCode)`：回填 exitCode、置 TERMINATED、启动缓冲 TTL 计时。
2. **兜底（attachedLate）**：插件晚装/晚启用时，用 `ExecutionManager.getRunningProcesses` 扫描既存进程补挂监听，并尝试从 ConsoleView Document 一次性抢救历史文本（标记 `attachedLate=true, firstAvailableOffset>0`，且抢救文本无 stream 标记——对 Agent 诚实）。

### 1.3 Agent 归属

MCP 工具 `run_configuration` 启动前把 `(token.subject)` 写入 ThreadLocal/环境的启动上下文，`processStarting` 回调中经 `env.executionId` 与"发起中启动"表匹配确定 owner；匹配不到的一律 `owner=user`。

## 2. 启动（run_configuration）

```kotlin
suspend fun start(project: Project, typeId: String?, name: String, debug: Boolean): ExecutionRecord {
    val settings = readAction {
        RunManager.getInstance(project).allSettings
            .filter { it.name == name && (typeId == null || it.type.id == typeId) }
    }.let { candidates ->
        when {
            candidates.isEmpty() -> throw NotFound(...)
            candidates.size > 1 && typeId == null -> throw InvalidArgument("同名配置歧义，请带 type_id：${candidates.map { it.type.id }}")
            else -> candidates.first()
        }
    }
    val executor = if (debug) DefaultDebugExecutor.getDebugExecutorInstance()
                   else DefaultRunExecutor.getRunExecutorInstance()
    // 冲突预检：单实例配置已有 running descriptor → CONFLICT（不让平台弹模态框）
    precheckSingleton(settings)
    return withContext(Dispatchers.EDT) {
        ExecutionEnvironmentBuilder.create(executor, settings).buildAndExecute()
        awaitRegistryRecord(executionId, timeout)   // processStarting/processNotStarted 双等待
    }
}
```

要点（均来自 R02）：

- **寻址**用 `(project, typeId, name)` 三元组；Spring Boot / Tomcat 等 Ultimate 闭源类型只按 `type.id` 字符串透传判别，不硬编码（实现期在真实 Ultimate dump 验证，写入文档）。
- 启动在 **EDT** 发起最稳；等待 record 出现用 `suspendCancellableCoroutine + withTimeout`。
- `wait` 语义在 record 之上实现：`exit`=等 processTerminated；`ready`=console 缓冲上做 `ready_pattern` 长轮询（复用 07 的 wait_for_pattern 机制）；到期返回 `state=timeout` + 已捕获 tail，不报错。
- before-launch 构建失败 → `processNotStarted` → 返回 `startupError` 并从 BuildService 附上构建诊断。

## 3. 停止（stop_session）

```
force=false: (handler as? KillableProcessHandler)?.killProcess()   // soft（给 Spring Boot 优雅停机机会）
             ?: handler.destroyProcess()
   → waitFor(timeout_sec) 不退出 → destroyProcess() force 兜底
force=true : 直接 destroyProcess()
Remote JVM Debug（attach 型）: detachProcess()，返回 detached=true 并注明
```

exitCode 从 `processTerminated` 回调拿（缓存在 record，Agent 事后可查——截图 3 的 `exit 137 SIGKILL` 场景即来源于此）。

## 4. 重启（restart_session）★核心①

| mode | 行为 |
| --- | --- |
| `same` | `withContext(EDT) { ExecutionUtil.restart(descriptor) }` —— 平台原生重启，等价于 Run 面板的 Rerun 按钮 |
| `run` / `debug`（与当前不同） | 编排：①stop（§3 流程）→ ②取 record.settings（手动会话从 `environment.runnerAndConfigurationSettings` 取）→ ③按目标 executor 走 §2 启动流程 |
| 会话已终止 | 跳过 stop 直接启动——"把刚退出的进程换 debug 再跑一遍" |

- 新 record 带 `restartOf=旧sessionId`，重启链可追溯；返回结构同 `run_configuration`，Agent 无需换心智。
- settings 为 null（极少数临时/外部进程）→ `NOT_FOUND` + remediation："该会话无关联运行配置，无法重启，请用 run_configuration 指定配置名"。
- 属主规则：自己的=operator；他人/user 的=admin + 审计（04 §4）。
- **验收场景**（用户核心需求）：IDE 里正在 run 的 `Application`（Spring Boot），Agent 一次调用 `restart_session(mode=debug, wait=ready, ready_pattern="Started")` → 旧进程优雅停止、以 debug 重启、断点即刻可用、返回就绪日志 tail。

## 5. 调试器控制（M4）

### 5.1 会话映射

`XDebuggerManager.getInstance(project).debugSessions` 与 ExecutionRegistry 经 `RunContentDescriptor`/ProcessHandler 关联：`ExecutionRecord.hasDebug=true` 时暴露 `debugSessionId`。

### 5.2 断点（breakpoints 工具）

- 查找 type：`XBreakpointUtil.findType("java-line")`（若该 util 属 impl 不稳定，回退 `XBreakpointType.EXTENSION_POINT_NAME` 遍历——R02 开放问题）。
- 增删：`XBreakpointManager.addLineBreakpoint / removeBreakpoint`，包 `WriteAction`（233~最新兼容的保险做法）；行号对外 1 基、平台 0 基，转换只在适配层。
- `canPutAt` 预校验并回报 `valid=false`（Agent 打在空行/注释上时立即知道）。

### 5.3 执行控制（debug_control）

- `pause/resume/stepOver/stepInto/stepOut/runToPosition` 直接映射 `XDebugSession` 同名方法（EDT 派发）。
- **关键工效设计**：每个动作后自动 `suspendCancellableCoroutine` 挂 `XDebugSessionListener.sessionPaused`（withTimeout）；命中后返回 `pausedAt{file,line,method,reason}` + 栈顶 5 帧——"step 然后看停哪了"一次往返完成。
- 超时未暂停 → 返回 `state=running`（如 resume 后不再命中断点，正常情况，不是错误）。

### 5.4 帧/变量/求值

- `computeStackFrames` / `computeChildren` / `evaluate` 全部是异步回调模型 → 统一 `suspendCancellableCoroutine + withTimeout` 桥接；**回调可能永不到达**（目标 JVM 死锁），一律硬超时返回结构化错误。
- 变量 `path` 下钻（`this.repo.items[3]`）：逐段 computeChildren 解析，每批 100（平台分页）透传 `hasMore`。
- `evaluate` 走 `XStackFrame.getEvaluator()`；admin + 三档策略 + 全量审计（04 §5）。

## 6. 边界与已知限制

| 边界 | 行为 |
| --- | --- |
| 同名多实例并行（allowParallel=true） | list_sessions 返回多条同 configName 记录，各自独立 sessionId |
| Gradle 委托运行 | ProcessHandler/console 结构可能不同（R02 开放问题），M1 验收含此场景实测 |
| PTY 会话 | stream 判别退化（stdout/stderr 混流），record 标记 `pty=true` |
| 用户在 IDE 里手动停止 Agent 启动的进程 | processTerminated 正常回调，Agent 下次读取时看到 TERMINATED+exitCode——无冲突 |
| IDE 关闭项目 | 该项目全部 record 置 TERMINATED，缓冲进入 TTL |

## 7. 热加载（HotSwap，M2）★核心③——debug 运行中免重启换入简单修改

用户心智：改完类 → 对该类"重新构建" → IDE 自动把新字节码换进正在 debug 的 JVM。`reload_classes` 把这条链路收敛为**一次 MCP 调用**。仅 debug 会话可用（JDWP `redefineClasses` 是调试器通道）。

### 7.1 平台 API（已对照 intellij-community master 源码核实，2026-07）

```java
// java/debugger/impl/src/com/intellij/debugger/ui/HotSwapUI.java
public abstract class HotSwapUI {
  public static HotSwapUI getInstance(Project project);
  public abstract void reloadChangedClasses(@NotNull DebuggerSession session, boolean compileBeforeHotswap,
                                            @Nullable HotSwapStatusListener callback);
  public abstract void compileAndReload(@NotNull DebuggerSession session, VirtualFile @NotNull ... files);
}
// java/debugger/impl/src/com/intellij/debugger/ui/HotSwapStatusListener.java
public interface HotSwapStatusListener {
  default void onSuccess(List<DebuggerSession> sessions) {}
  default void onNothingToReload(List<DebuggerSession> sessions) {}
  default void onFailure(List<DebuggerSession> sessions) {}
  default void onCancel(List<DebuggerSession> sessions) {}
}
```

### 7.2 执行流程

```
reload_classes(session_id, files?)
 1. record = ExecutionRegistry.find(session_id)；record.hasDebug == false → NOT_DEBUG_SESSION
    （advice："run 模式无法热加载，可 restart_session(mode=debug) 换 debug 跑"）
 2. DebuggerSession 映射：DebuggerManagerEx.getInstanceEx(project).sessions 中
    process.processHandler == record.processHandler 者
 3a. files 指定：ProjectTaskManager.compile(vfiles).await()（⇧⌘F9 语义，复用 05 双通道诊断）
       编译失败 → outcome=failed + compile.errors，不发起重载
       编译成功 → HotSwapUI.reloadChangedClasses(session, compileBeforeHotswap=false, listener)
 3b. files 缺省：reloadChangedClasses(session, compileBeforeHotswap=true, listener)
       （等价 IDE"构建项目 → 重载全部已修改类"）
 4. listener 四回调 → suspendCancellableCoroutine + withTimeout 桥接 → outcome
 5. 审计（operator；属主规则同 stop/restart）
```

编译单独走 3a 的原因：`compileAndReload` 无 listener 重载——组合"compile→reload"才能同时拿到**结构化编译诊断**与**热替换结果**。

### 7.3 能力边界（与用户确认的"简单修改"定义一致）

| 修改类型 | 结果 |
| --- | --- |
| 方法体内部改动（逻辑/常量/局部变量） | ✅ `reloaded`，进程 pid 不变，新逻辑立即生效 |
| 方法/字段增删、签名变更、类增删、继承结构变化 | ❌ `failed`（JDWP redefineClasses 标准限制）+ advice："结构性变更需重启：restart_session(mode=debug)" |
| 无改动 | `nothing_to_reload` |
| run（非 debug）会话 | `NOT_DEBUG_SESSION` + advice |

### 7.4 实现期核对清单

1. HotSwapUI / HotSwapStatusListener 位于 java debugger **impl** 模块（包名 `com.intellij.debugger.ui`，非 openapi）→ 收敛进薄适配层，每版本回归；
2. 弹窗抑制：Settings→构建→调试器→HotSwap 的 `RUN_HOTSWAP`(ask/always/never) 对程序化调用路径是否仍可能弹确认框需实测；必要时调用期临时置 always 并恢复；
3. `onFailure` 不携带逐类失败原因（详情走 IDE 通知/HotSwapProgress）→ v1 返回 outcome+advice，逐类失败明细列为开放问题；
4. 同项目多个 debug 会话并存时，reloadChangedClasses 按传入 session 定向，验证不误伤其他会话；
5. Kotlin 类的 HotSwap 行为（inline/lambda 编译产物）需专项用例；
6. DCEVM / JetBrains Runtime 增强热替换（放宽结构性变更）→ 探测到时可放宽 7.3 边界，列观察项（09）。
