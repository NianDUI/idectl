## 已完成（控制台缓冲增强 + 整包重命名，2026-07-04）

**动机**：用户三问——①「获取控制台内容效率高么」②「IDEA 控制台老内容被刷没，MCP 有这问题么」③「搜索够用么/重启前日志还在么」。答复要点：控制台采集是 `ExecutionRegistry.attach` 在 `ProcessHandler` 上挂 `onTextAvailable` **独立采集**（不经 IDEA 控制台 UI），故不继承 UI cycle buffer「翻不到老内容还不自知」的毛病；但内存环自身有上限（默认 10k 行/8MB），淘汰时**显式报 `gap+droppedLines`**（Kafka 语义）。据此做两件事：

1. **控制台缓冲可配置**（设置页新增四项，`BridgeSettings` 持久化）：内存缓冲行数、内存缓冲上限(MB)、启用磁盘归档、磁盘归档上限(MB)。改动只对**新启动**会话生效（不动运行中会话）。`ExecutionRegistry.consoleConfig(sessionId)` 每次启动从设置构建 `ConsoleConfig` 注入 `ConsoleStore`。
2. **磁盘归档 `ConsoleArchive`**（tier 2，让被淘汰老日志仍可 `console_read`/`console_search` 回溯）：
   - `ConsoleRingBuffer.evict()` 淘汰前先 `archive.append(removed)` 落盘；`read()` 跨「归档段(磁盘)→内存环」两层回填；`snapshot(includeArchiveFrom)` 供搜索把归档历史并入 grep 视图（`ConsoleSearchTool` 改用 `record.console.searchSnapshot(fromOffset)`，上限 100 万行护栏）。
   - **省内存设计**（关键）：`ConsoleArchive` **无内存索引**——读取时开临时 `RandomAccessFile` 在 ≤8MB 段内**顺序游走定位**（读少、段有界，成本可接受），故开启归档后堆开销 ≈ 0；**只保留 1 个打开的写句柄**（当前段），旧段封存即关流；总量 `maxBytes` 有上限，超了删最旧段并推进 `archiveFirstOffset`（读越界照样报 gap）。
   - **清理**：`ConsoleStore.dispose()` 关归档删临时目录；`ExecutionRegistry.scheduleEviction`（TTL 30min 到期）与 `disposeAll()` 调用；`BridgeProjectService` 实现 `Disposable`，项目关闭时 `executions.disposeAll()` 清全部临时文件。临时目录 `PathManager.getTempPath()/idea-bridge-console/<sessionId>`。
   - **默认开启**（用户点拨「归档默认开更省内存」——正确）：归档不增加常驻内存（只把淘汰行从 RAM 挪到磁盘），配合精简内存环即可「留存历史而不撑大 RAM」；无索引设计使其堆开销 ≈0，故 `consoleArchiveEnabled` 默认 `true`，代价仅为有上限、会话结束即清的磁盘。

**搜索够用么/重启前日志**（答复已落文档）：`SearchEngine` = `grep -E -i -A -B -m` + offset/时间/stream 过滤 + 500ms ReDoS deadline；短板仅「单行匹配」(用 `-A/-B` 上下文补)与「搜不到被淘汰老日志」——后者本次归档已补。`restart_session` 起**全新会话**，旧日志挂**旧 sessionId** 保留 30min（返回 `restartOf` 指向旧 id）；归档开启后旧会话磁盘归档在 TTL 内同样可回溯。

## 运行时踩坑记录（runIde 实测）

1. **`@Service` 漏注解**（2026-07-04，已修）：`BridgeSettings` 只写了 `@State` 没写 `@Service(Service.Level.APP)`，runIde 启动时 `appFrameCreated → IdeaBridgeService` 构造里 `service<BridgeSettings>()` 报 `Cannot find service`，整条启动链断、server 未起。→ 补 `@Service(Service.Level.APP)`（PersistentStateComponent 可同时是 @Service）。教训：本项目所有用 `service<T>()`/`project.service<T>()` 取的类都必须有 `@Service` 注解（IdeaBridgeService/AuditService/BridgeProjectService/BridgeSettings）。
2. runIde 日志里大量 "Plugin ... has dependency on 'JetBrains Ultimate' which cannot be loaded"（Spring/Tomcat/JPA 等）——`intellijIdea` 无 Ultimate license 时的**正常现象**，Community 级 SDK 忽略即可。
3. `DaemonRequestException: ... does not contain product-info.json` + `cwm-plugin` 加载失败——远程开发/CWM 守护发现的报错，**非致命**，不影响本地 IDE frame 与插件。
5. **★VFS 未刷新导致 build/reload 空转**（2026-07-04，已修，对整个产品关键）：Agent 用文件工具**从外部改磁盘**后调 `reload_classes` → `outcome=nothing_to_reload`（compile ok）。根因：IDE 的 VFS 不知道文件变了，JPS 认为无改动 → 编不出新 .class → 没东西热替换。IDE 里出现"外部文件更改同步可能缓慢 / 无法接收文件系统事件通知"警告印证了这点（本项目文件监视器收不到 FS 事件）。窗口聚焦（IDE 默认帧激活时同步）能临时触发刷新，实测聚焦后 reload 立刻 `reloaded` 且端到端生效（curl 响应变新、进程 pid 不变）。→ 修法：新增 `core/util/refreshProjectVfs(project)`（`VfsUtil.markDirtyAndRefresh(async=false, recursive, reloadChildren, projectRoot)`，off-EDT），在 `HotSwapService.reload` 与 `BuildService.build` 编译前调用。**已真机验证**：重启 runIde 后，外部改文件→`reload_classes`**免聚焦窗口**即 `reloaded`、curl 响应变新、pid 不变。教训：**凡是 Agent 外部改文件后要 build/编译/热加载，必须先同步刷新 VFS**。后续优化：只刷改动的文件而非全项目（大项目全刷会慢，正是那个警告说的）。IDE 自带的"应用热交换"弹窗与本工具功能重叠但无需交互（程序化 reloadChangedClasses 不依赖它）。

4. **★Ktor classloader 冲突**（2026-07-04，已修，最重要）：运行时报 `io.ktor.server.application.Application cannot be cast to io.ktor.util.pipeline.Pipeline`（分属 PluginClassLoader 与 PathClassLoader）。根因：**平台 2026 自带 Ktor 3.x**（`lib/intellij.libraries.ktor.server.cio.jar` 等，含完整 server/routing/engine/request/response），而我捆绑了 Ktor **2.3.13** → 双 classloader 类图割裂。→ 修法：build.gradle 里 Ktor **不再声明任何依赖**（平台的 ktor jar 已在编译+运行时类路径上），零捆绑，全从平台单一 classloader 加载。我的路由 helper 都显式接 `ApplicationCall`、不碰 2.x/3.x 的 PipelineContext↔RoutingContext 差异，所以对平台 3.x 直接编译通过。产物 zip 现在只剩插件自身 jar。
   - 教训：**插件里任何第三方库都要先查平台 `lib/` 是否已自带**（`ls .../transformed/idea-2026.1.4/lib | grep <lib>`）；自带就 compileOnly/零捆绑，否则 classloader 冲突。
